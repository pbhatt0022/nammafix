# NammaFix AI — Decisions & Rationale

> Why the project is the way it is. Lightweight ADR log. Newest at top.
> If you're an agent about to "improve" something here, read the rationale first —
> several of these are deliberate and reversing them re-introduces known risk.

---

## D-012 · Translate data enums via keyed lookups, not a value map
**Decision:** Category / status / severity values (which arrive as English enums from the backend)
are translated in the UI with `t("enum.cat."+value)`, `t("enum.status."+value)`, `t("enum.sev."+value)`.
**Why:** The enum strings double as both data and display. Keying translations off the raw value
keeps one source of truth, lets the same badge translate everywhere, and needs no mapping table.
Backend stays English; only the rendered label changes.

## D-011 · Retone the dashboard by remapping Tailwind ramps in `@theme`, not per-file
**Decision:** The dashboard was authored in cold `slate/indigo/emerald`. Instead of hand-editing
~300 class usages, the cold ramps are **redefined to warm civic tones inside `@theme`** in
`src/index.css`. The landing uses the civic tokens (`cream`/`ink`/`indigoc`), not these ramps, so
it's unaffected.
**Why:** One reversible place retones the whole app; shortest, lowest-risk path to cohesion under a
deadline. Muted label color was darkened to clear WCAG AA (~4.8:1).
**Reconsider if:** a component needs a genuinely cold neutral — then use an explicit hex, not a ramp.

## D-010 · Voice input via browser-native Web Speech API (not Gemini audio)
**Decision:** Voice-to-text on the report description/landmark + verification comment uses the
native `SpeechRecognition` API (`MicButton.tsx`); recognition language follows the selected UI
language. Gemini audio transcription was the alternative.
**Why:** Web Speech is free, needs no key, and — crucially — **doesn't spend the Gemini quota**
(see D-009 quota note). In Chrome it's Google's own speech recognition and supports Indian
languages. Trade-off: Chrome/Edge-first, partial Safari, absent in Firefox — the button hides
itself when unsupported.

## D-009 · Multilingual = static Gemini-generated UI strings + live Gemini for content
**Decision:** Ship a scoped i18n: (1) **static** translation of UI strings — English is the source
(`en.json`), the other 6 languages are **pre-generated by Gemini** into `translations.json` via
`scripts/gen-i18n.mjs` and committed; runtime makes **zero** translation calls. (2) **Live** Gemini
translation only for dynamic AI content (`/api/translate` + `TranslatePacket`). Languages: en, hi,
kn, ta, te, bn, mr. Engine: **Gemini** (already configured). **Rejected** the unofficial
google-translate proxy (ToS-violating, IP-blocked from server IPs, fails mid-demo) and a separate
Google Cloud Translation API key (extra billing/setup). Bhashini (Govt NLTM) is the *production*
story, not built.
**Why:** Static strings can't fail in front of judges and cost nothing at runtime; live Gemini is
reserved for the genuinely dynamic text. "Full multilingual" was on the MVP *don't-build* list, but
this scoped form is reliable and strengthens the Innovation + Google-tech scores.
**Quota caveat:** the Gemini free tier is **20 requests/day** on the current `AQ.` key. UI
translation is unaffected (static), but every live AI call (packet, copilot, closure, `/api/translate`,
re-running `gen-i18n.mjs`) spends quota. **Enable billing / use an `AIzaSy…` key before judging.**

## D-008 · Auto-routing via a MOCK resolver directory, not real govt integration
**Decision:** On report create, `routeReport()` maps category → department (BBMP/BWSSB/BESCOM…) via
a hardcoded `RESOLVER_DIRECTORY`, resolves a ward via `lookupWard()` (Bangalore lat/long bounding
boxes), and emits a `RoutingTicket` (dept, division, ward, channel, SLA, dispatch note). Critical
issues get a halved SLA. Surfaced as a "Civic Dispatch Ticket" in the case view.
**Why:** Demonstrates the "identify where to report → raise a ticket → route it" agentic step
cheaply and demo-stably. The MVP plan's honest wording — "mock resolver directory / resolver-ready
routing note," NOT "integrated with municipal systems." UI labels it as a mock. Production path is
live Google Maps geocoding against a municipal jurisdiction DB.
**Reconsider if:** demoing outside Bangalore — the ward boxes fall back to a default zone.

## D-007 · Docs live in `/docs`, README stays at root
**Decision:** All project markdown except `README.md` lives in `/docs`. README stays at repo root.
**Why:** GitHub renders the root README on the repo landing page; moving it breaks that convention
and the hackathon expects a discoverable README. Everything else (plan, decisions, state, MVP spec)
is reference material and belongs together in `/docs`.

## D-006 · Strengthen duplicate detection; keep it rule-based
**Decision:** Add a real `checkDuplicates()` that runs on report submit, using category match +
landmark/geo proximity + keyword overlap. Do NOT use a vector DB or an LLM-judge for it.
**Why:** Duplicate handling is a named example feature and feeds the 20% Agentic Depth score.
Rule-based is cheap, explainable, and demo-stable. Vector/LLM approaches add cost and flakiness for
no judging upside at this scale.
**Status:** ✅ Built — `checkDuplicates()` runs on submit (Haversine distance + category + keyword
overlap); the report form surfaces a "similar issue(s) nearby" warning.

## D-005 · In-memory data store, seeded — no Firebase, no JSON/SQLite
**Decision:** Keep the in-memory `database` object in `server.ts`, seeded from
`src/data/mockReports.ts`. No external persistence.
**Why:** The original MVP plan suggested Firebase, but on a 5-day clock with a mandatory Google AI
Studio deployment, adding a database is multi-hour risk for little judging value. Seed data always
reappears on boot, so the demo is fully populated and stable. Trade-off: reports created during
judging don't survive a cold start — acceptable for a demo, and called out honestly.
**Reconsider if:** deployment requires persistence, OR judges are told data persists. On Cloud Run,
a JSON-file store may not survive the ephemeral filesystem anyway — verify before relying on it.

## D-004 · Hero illustration is SVG; raster (Higgsfield) is an additive swap
**Decision:** The hero is hand-drawn inline SVG (`DoodleHero.tsx`). A generated raster can be
dropped in by setting `heroImage` in `landing/assets.ts`; the SVG stays as the permanent fallback.
**Why:** SVG is on-palette, themeable, weightless, and unblocked. Higgsfield generation was blocked
(the OAuth account returns "User not found" — no provisioned Higgsfield user). User explicitly wants
the SVG kept even after trying Higgsfield, so the swap must never delete the SVG branch.

## D-003 · Landing page fronts the existing app via a view switch
**Decision:** `Root.tsx` toggles between the new `Landing` and the existing `App` dashboard.
Landing CTAs deep-link via an `onEnter(intent)` callback (report vs. map); the app's logo returns
home. `App.tsx` gained optional props (`initialTab`, `startReporting`, `onHome`) whose defaults
preserve the original behavior.
**Why:** Adds the landing page with zero changes to app/backend logic and no router dependency —
shortest, lowest-risk integration.

## D-002 · Visual identity: "Bharat Civic Folk-Tech"
**Decision:** Warm Indian civic palette (turmeric cream + civic indigo base; saffron/marigold/
peacock/lotus/leaf accents), Anek/Mukta/Honk type, hand-drawn ink-outline doodles, municipal
docket/stamp UI motifs. Folk art is an *accent layer*, never dense behind body text.
**Why:** Differentiates from generic SaaS / Western 311 clones; "Indian-first, civic-first" identity
that's memorable in 3 seconds. Tokens live in `index.css` `@theme`. Avoids religious/political/
nationalistic imagery by design.

## D-001 · Gemini model = `gemini-2.5-flash`, multimodal, JSON outputs
**Decision:** Single model `gemini-2.5-flash` for all four AI functions (issue analysis, resolver
copilot, closure verification, impact insight). Key from `GEMINI_API_KEY`; falls back to a
SIMULATED mode when absent.
**Why:** Flash is fast and cheap enough for live demo, multimodal (image+text), and supports the
1M-token context. Simulated fallback keeps UI work unblocked without burning quota and protects the
demo if the key/quota fails. **Open risk:** malformed JSON from the model — needs parser fallbacks
(see PLAN P1 #6).

### Things deliberately NOT done
Real municipal/government API integration · MCP servers / agent runtimes / Agent Gateway / Sandbox ·
Firebase (for now) · legal-admissibility or cryptographic evidence claims · advanced RAG / vector DB ·
public ward shaming rankings · the unofficial google-translate web proxy (see D-009) ·
Google Cloud Translation API as a separate key (Gemini covers it).
Rationale: out of scope for a 5-day MVP and/or overclaiming; the MVP plan excludes most of these.

> Shipped *scoped* versions of two things the MVP plan listed as "don't build": **multilingual UI**
> (static + live, D-009) and **voice** (Web Speech dictation, not full voice-first, D-010).
